# Modes
- I think we might be looking at a few different "modes" the tools we are developing here can run in.

> **[Claude]:** This framing is clarifying, not muddying. "Modes" is doing real work here — the security model, the fidelity of data exposure, the infrastructure required, and the intended consumer (human dev vs. AI vs. automated pipeline) are all different across the three. Naming them is the right move before we go any further on implementation. One reframe worth considering: these are less "modes of the tool" and more "deployment contexts" — the tool adapts to the context it's running in rather than the developer toggling a mode switch. That matters for architecture: you don't want a boolean `mode: 'exploratory' | 'test' | 'lint'` in a config object, you want each context to compose in only what it needs.

## Exploratory Dev Mode

  - An "exploratory" mode - which is in an app, at development time, installing the stellarDevTools and using those together (developer, AI) to gain insights into the application, track down issues, etc. I think the state viewer for the signal store is a key part of this. I also think we are going to have to offer a similar thing for the NGRX/store (the redux style), but maybe that is a "plugin" (see part 1 of the overview.md). So I'm wondering if we modularize this even further than we were initially thinking.

So, app.config.ts could turn into something like:

```ts

  useStellarDevTools(
    withNgrxSignalStoreInsights(),
    withNgrxReduxStoreInsights(),
    withHttpTrafficInsights(),
    // etc. etc.
  )
]
```

The more important naming question is `useStellarDevTools` vs `provideStellarDevtools` — the existing docs use `provideStellarDevtools`, which matches Angular's own `provideRouter()` / `provideHttpClient()` convention. Two separate plugins with their own configs is cleaner even if the first version ships with only one of them.

## Testing Mode

I also think there are going to be some aspects of this that will have to run within tests. I think in a test environment (like Playwright) we can more confidently give you, the llm more access to what is happening in the browser, including even giving you access to the DOM, etc.

> **[Claude]:** The key architectural consequence here is that test mode doesn't need sanitization at the same level — the state being inspected is test fixture data, not production PII. That said, I'd resist removing sanitization entirely even in test mode; teams run tests against anonymized production data more often than they admit. The safer design is "sanitization opt-out is explicit and scoped to the test environment" rather than "test mode skips sanitization by default."
>
> The Playwright angle is interesting because it inverts the direction: instead of Stellar pushing data to a surface I can read, Playwright gives me a handle to pull from the browser directly. `page.evaluate(() => window.__stellarDevtools.snapshot())` works today with no additional infrastructure. That's worth documenting as a supported pattern — it's the zero-cost version of the MCP integration for test contexts.

## Runtime Lint Mode

This is maybe a subset of the testing mode, but I'm thinking there might be big value in us defining and creating a way to capture use in an application (like Tea Leaf, but lighter, less fidelity would be needed. More 'User entered here, navigated here, clicked here, etc.)' and having that in a format that you (AI) could generate a playwright test or series of tests to replicate the user session.

I think we could, over time, develop some sort of rubrics on things that might trigger investigation for optimizations on the app. Everything from repaints, to optimizing data access through APIs, and (for Angular) a huge thing would be in optimizing service creation and destruction for memory issues, data "staleness and freshness", and detecting accidental duplication of created services (that are provided in multiple places).

> **[Claude]:** This is the causal event stream from the `overview.md` backlog, arriving from a different angle. The session capture idea (user interactions → structured event log → Playwright test generation) is exactly what `withPlaywrightObserver()` was sketched for. Worth linking those explicitly so the design doesn't diverge.
>
> The "rubrics" idea is the most ambitious part and also the most valuable if done right. Angular-specific heuristics (service duplication, premature destruction, change detection thrashing) are the kind of thing that currently requires a senior dev to know what to look for. Encoding them as detectable patterns that can be flagged automatically — and then handed to me with context — is genuinely novel. The causal event stream is the prerequisite: you can't detect "service created twice" without a timeline.
>
> One flag: "lint" as a name might undersell this. Static linters check code; this checks *runtime behavior*. "Runtime diagnostics" or "behavioral analysis" might be more accurate and set better expectations for what it can and can't catch.

>> Jeff: Rubrics isn't quite the right word, but I have a suspicion that this kind of feature is the real spirit of what we are trying to do here. So, not just simple "the application blows up when you do this, fix it" type stuff, but more the human and the AI collaborating - the human helping to make the "felt sense" of the application more legible to the AI ("it feels sluggish when traversing this route", or "a bit janky when expanding this drop down"), finding ways to put some metrics around things like that, and finding alternative "right" ways to avoid those kind of things. It is ambitious - but I really want to explore the space of emphasizing using AI to make *better* software, not just more of it (I think we have a little too much already, to be honest, and there are a hundred thousand folks out there having you throw more onto the pile every day!).

> **[Claude]:** The security gradient you're describing maps cleanly: exploratory mode = sanitization mandatory, data is production state; test mode = sanitization opt-out allowed, data is fixture state; lint mode = no state values at all, only structural/behavioral signals (service IDs, event counts, timing). Lint mode is actually the most privacy-safe by default because it never needs to see state *values* — only that a service was created, destroyed, how many times, in what sequence. That's a useful property to lean into when designing the lint data model.
